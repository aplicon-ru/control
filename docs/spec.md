# Универкон Control — Спецификация продукта
**Версия:** 0.5 (рабочая)  
**Статус:** обсуждение не завершено  
**Дата:** 2026-07-02

---

## 1. Концепция

**Универкон Control** — единая операционная консоль экосистемы Универкон. Заменяет отдельные admin UI каждого модуля, SSH-доступ к серверам и ручное управление Docker. IT-персонал университета работает только через CP.

### Три формы одного инструмента

```
Один Go-бинарник ukon

ukon --ui --port 8080       → Web UI (основной)
ukon server add --host ...  → CLI (для продвинутых)
curl install.sh | bash      → тонкий скрипт-обёртка над CLI
```

## 2. Топология

```
┌─────────────────────────────────┐
│        Control Plane (CP)       │
│        отдельный сервер         │
│                                 │
│  Web UI · CLI · API             │
│  SQLite · master.key            │
│  Update watcher · Notify hub    │
└──────────────┬──────────────────┘
               │ SSH (outbound)
    ┌──────────┼──────────────┐
    ▼          ▼              ▼
[Server A]  [Server B]   [Server C]
prod        staging      remote-uni
docker      docker       docker
               │
               └──── heartbeat ──► Лицензикон API
                    (через CP, не напрямую)
```

### Типы целевых серверов

| Тип | Описание |
|---|---|
| **docker-only** | CP управляет только Docker-стеком. ОС — чужая зона |
| **full** | Сервер полностью отдан CP. CP следит за ОС, обновлениями, безопасностью *(не MVP)* |
| **mock** | Псевдосервер для демо и обучения. SSH не используется, операции симулируются |

CP помечает: `[THIS]` — сервер где запущен сам CP (нельзя удалить и остановить CP через него), `[MOCK]` — mock-сервер.

## 3. Авторизация и роли

### Вход

```yaml
auth:
  local: true            # всегда доступна как fallback
  oidc:
    enabled: true
    provider: keycloak   # или google, azure, univerkon, любой
    client_id: "..."
    client_secret: "..."
    issuer_url: "https://auth.medu.ru/realms/university"
```

Один OIDC-провайдер на инстанс CP. Собственная авторизация всегда доступна параллельно.

### Защита доступа

- 2FA (TOTP) — опционально, настраивается per-user
- IP whitelist — разрешённые адреса для входа
- Оба можно включить одновременно

### Роли

| Роль | Возможности |
|---|---|
| **Super Admin** | Всё: пользователи, организации, квоты, глобальные настройки CP |
| **Org Admin** | Все серверы своей организации: деплой, лицензии, брендинг, S3, логи |
| **Operator** | Деплой и обновления на назначенных серверах. Без удаления и настроек |
| **Viewer** | Только чтение: статусы, логи, лицензии |

Сервер привязан к организации. Org Admin видит только свою организацию.

## 4. Первый запуск — Wizard

Минимально достаточный для старта набор шагов:

1. Создать Super Admin (login + password + опционально 2FA)
2. Настроить OIDC (или пропустить — работать через local auth)
3. Добавить первый сервер (хост, SSH-ключ/пароль, тип: docker-only / full / mock)
4. Назначить S3 для бэкапов CP
5. Готово → Dashboard

## 5. Домены

### 5.1 Server Registry

Что хранит CP по каждому серверу:

```yaml
server:
  id: 1
  name: prod-medu-01
  host: 192.168.1.10
  port: 22
  user: deploy
  auth: key              # key | password
  type: docker-only      # docker-only | full | mock
  org: medu-orb
  environment: prod      # prod | staging | dev
  tags: []               # зарезервировано, не MVP
  installed_modules: [univerkon-core, testikon, didaktikon]
  s3_pool: [minio-local, s3-external]
```

### 5.2 Module Registry

Каждый модуль = `module.yaml`:

```yaml
id: testikon
name: Тестикон
version: 1.2.0
image: ghcr.io/aplicon-ru/testikon:1.2.0
compose: ./modules/testikon/compose.yml
vars_schema: ./modules/testikon/schema.json
healthcheck: http://localhost:8001/health
deps: [univerkon-core]
```

Реестр версий — GitHub Releases или собственный manifest.json по URL.

**Конфигурация модулей через CP:**
- schema-driven форма (из `vars_schema`)
- Изменения не применяются мгновенно — оператор видит diff и нажимает «Применить»
- CP предлагает сделать бэкап конфигов перед применением
- Drift detection: CP видит расхождение между конфигом в CP и тем что реально на сервере

### 5.3 Deployment Engine

**Pre-flight checks перед деплоем:**

| Проверка | Уровень | Минимум | Рекомендовано |
|---|---|---|---|
| Свободное место `/` | BLOCK | 2 GB | 10 GB |
| Свободное место `/var/lib/docker` | BLOCK | 5 GB | 20 GB |
| Свободная RAM | BLOCK | 512 MB | 2 GB |
| CPU load average | WARN | — | < 0.8 per core |
| Swap настроен | WARN | — | ≥ 1 GB |
| Docker установлен и daemon запущен | BLOCK | — | — |
| docker compose v2 доступен | BLOCK | — | — |
| Версия Docker ≥ минимальной из `module.yaml` | BLOCK | — | — |
| Права на docker socket | BLOCK | — | — |
| Порты модуля свободны | BLOCK | — | — |
| Доступность реестра образов (ghcr.io) | BLOCK | — | — |
| DNS резолвится | BLOCK | — | — |
| Домен резолвится на этот сервер (HTTP-01) | WARN | — | — |
| Все зависимые модули установлены и healthy | BLOCK | — | — |
| Версии зависимостей совместимы | BLOCK | — | — |
| Назначенный S3 доступен и права на запись | BLOCK | — | — |
| Лицензия активна | INFO | — | Полный режим |
| Лицензия отсутствует или истекла | WARN | — | Демо-режим |
| Лицензия истекает < 30 дней | WARN | — | Напоминание |
| Сертификат истекает < 14 дней | WARN | — | — |
| Порт 80 открыт (HTTP-01 валидация) | WARN | — | — |

BLOCK — деплой не запускается. WARN — оператор видит предупреждение и подтверждает продолжение.

```
Deploy flow:
  → maintenance page → nginx push
  → docker compose pull
  → docker compose up -d
  → healthcheck
  → OK: убрать maintenance page
  → FAIL: rollback → убрать maintenance page → алерт

SSE-стриминг лога в UI и в CLI stdout
```

**Maintenance page:**
- Текст и внешний вид в стиле брендинга организации
- Оператор задаёт ожидаемое время работ
- Шаблон настраивается в разделе Брендинг

**Групповые операции:**
Деплой / обновление сразу на несколько серверов. Результат — статус по каждому отдельно.

**Rollback:**
CP хранит предыдущую версию. Кнопка «Откатить» доступна после каждого обновления.

**История операций:**
Полная история деплоев с логами. Зашёл через месяц — увидел что, когда, кем. Лог той сессии сохранён.

**Шаблоны:**
Сохранить набор модулей + брендинг + конфиги как шаблон. Новый университет → выбрал шаблон → задеплоил.

**Клонирование конфига:** не планируется.

### 5.4 License Manager

```
Жизненный цикл лицензии:

  купил ключ
    → ввёл в CP
    → CP активирует через Лицензикон API
    → получает JWS токен
    → пушит токен на сервер при деплое
    → модуль валидирует JWS локально (без интернета)
    → CP периодически опрашивает Лицензикон API (pull)
    → строит dashboard: серверы × модули × статус
```

Что хранит CP:

```yaml
license:
  key: "XXXX-XXXX-XXXX-XXXX"
  module: testikon
  server_id: 1
  org: medu-orb
  issued_at: 2025-01-01
  expires_at: 2026-01-01
  status: active       # active | expiring_soon | expired | revoked
  last_heartbeat: 2025-07-01T12:00:00Z
```

**Алерты:** 30д → 14д → 7д → 3д → истекла.

**Демо-режим:**
Модуль без лицензии (или с истёкшей) деплоится и работает в демо-режиме. CP при деплое пушит флаг `demo: true` в конфиг модуля. Модуль сам применяет ограничения согласно своему `module.yaml`:

```yaml
demo_mode:
  limits:
    users: 10
    storage_gb: 1
    features: [basic]
  banner: true        # показывать баннер «Демо-версия»
  expiry_days: 30     # null = бессрочно
```

Ограничения демо-режима определяет разработчик каждого модуля в его `module.yaml`. CP не знает про конкретные ограничения — только про флаг.

**Перенос лицензии:**
CP отзывает JWS у Лицензикона → получает новый → пушит на новый сервер. IT-персонал — только выбирает откуда/куда.

### 5.5 Update Watcher

```
Cron (настраиваемый интервал, по умолчанию 6ч):
  → скачивает manifest.json из реестра
  → сравнивает с installed версиями на всех серверах
  → создаёт запись UpdateAvailable
  → уведомляет через Notification Hub
  → в UI: badge на сервере и модуле

Оператор:
  → видит "Доступно: 1.2.0 → 1.3.0 · Changelog"
  → нажимает Apply (или назначает расписание)
  → стандартный deploy flow
```

Auto-update — нет. Только уведомление + ручное применение.

**Расписание операций:**
Деплой / обновление можно назначить на время (например 03:00). CP ставит задачу и выполняет по расписанию. Maintenance page включается автоматически.

### 5.6 Notification Hub

Три канала:

| Канал | Протокол |
|---|---|
| Telegram | Bot API |
| Max (VK) | Bot API (`botapi.tamtam.chat`) |
| Email | SMTP |

Приоритеты настраиваются per-событие:

```yaml
notifications:
  channels:
    telegram:
      bot_token: "..."
      chat_id: "..."
    max:
      bot_token: "..."
      chat_id: "..."
    email:
      smtp_host: mail.medu.ru
      from: noreply@medu.ru
      to: [admin@medu.ru]

  events:
    license_expiring_30d: [telegram]
    license_expiring_7d:  [telegram, email]
    license_expired:      [telegram, email, max]
    update_available:     [telegram]
    deploy_failed:        [telegram, email]
    deploy_success:       []
    server_offline:       [telegram, email, max]
    backup_failed:        [telegram, email]
    backup_invalid:       [telegram, email]
    cert_expiring_14d:    [telegram]
```

### 5.7 S3 / MinIO

**Пул S3:**
Несколько источников — managed MinIO и внешние — в общем пуле. Каждому модулю назначается конкретный S3.

```
Mode A — Managed MinIO:
  CP разворачивает MinIO как модуль (docker compose)
  CP создаёт бакеты под каждый модуль
  CP раздаёт credentials как Class C секреты

Mode B — External S3:
  Указываешь endpoint + credentials
  CP проверяет доступность и права
  Совместимые: AWS S3, Яндекс, Selectel, Cloudflare R2
```

**Файловый менеджер:**
Встроен в CP. Org Admin видит только бакеты своей организации. Operator — те бакеты к которым у него есть доступ.

**SSL / домены:**
CP управляет TLS на себе и на целевых серверах.
- Let's Encrypt — HTTP-01 (CP сам ставит валидационный файл)
- Собственный сертификат — загрузка в CP (включая ГОСТ-сертификаты)

### 5.8 Диагностика и Мониторинг

**Диагностика сервера:**
- Свободное место / RAM / CPU
- Версия Docker
- Открытые порты
- Статус контейнеров
- Последние ошибки в логах

**Логи модулей:**
`docker logs` нужного контейнера из CP. Поиск и фильтр по времени.

**Мониторинг ongoing:**
- Uptime модулей
- Падения и перезапуски контейнеров
- Агрегация по организации (суммарные данные по всем серверам)

Всё в единой панели мониторинга внутри CP.

**Публичная статус-страница:**
`status.domain.ru` — автоматически генерируется CP. Брендируется под организацию. Два состояния: «Все системы работают» / «Ведутся технические работы». Плюс баннер в LMS.

### 5.9 Hardening сервера (при добавлении)

Опциональный чеклист при добавлении нового сервера:
- Отключить root login по SSH
- Настроить fail2ban
- Настроить ufw (только нужные порты)

Каждый пункт — галочка, можно применить выборочно.

### 5.10 Аудит лог

Каждое действие в CP фиксируется: кто, что, когда, на каком сервере. Включая использование секретов.

### 5.11 Webhooks *(не MVP)*

Внешние системы получают события от CP через webhook. Обсудить отдельно: список событий и формат payload.

### 5.12 Квоты *(не MVP)*

Org Admin управляет N серверами и M лицензиями. Super Admin задаёт квоты на организацию.

## 6. Архитектура секретов

### Три класса

```
Class A — CP Secrets
  Telegram/Max bot token, SMTP password,
  OIDC client secret, Лицензикон API key

Class B — Server Credentials
  SSH private keys, sudo passwords

Class C — Module Secrets (пушатся на сервер при деплое)
  Пароли БД, API ключи, S3 credentials, JWS токены
```

### Envelope Encryption

```
При первом запуске CP:
  → генерирует master.key (256 бит)
  → master.key — отдельный volume, отдельный путь
  → опционально защищён passphrase (запрашивается при старте)

Все секреты в SQLite:
  → зашифрованы AES-256-GCM через master.key
  → в логах и SSE stream — маскируются (****)
  → в бэкапе — архив и master.key хранятся раздельно
```

```
docker run \
  -v ./data:/data     # SQLite, конфиги
  -v ./keys:/keys:ro  # master.key отдельно
  ghcr.io/aplicon-ru/control
```

### Секреты на целевом сервере

```
НЕ так: environment: DB_PASSWORD: mysecret  (в compose.yml)

ТАК:
  .env файл (chmod 600, owner root)
  compose.yml ссылается на .env
  CP пушит .env через SSH, не хранит открыто
```

### SSH ключи

При добавлении сервера:
- Загрузить существующий ключ
- CP генерирует пару сам → показывает публичный ключ один раз → «добавьте в authorized_keys»

Приватный ключ после сохранения — только зашифрованным. Показать повторно нельзя, только заменить.

## 7. Backup

### Объекты

| Объект | Что | Куда |
|---|---|---|
| CP | SQLite + конфиги (без master.key) | S3 из пула |
| Managed сервер | Приложение целиком: образ версии + docker volumes + конфиг | S3 из пула |
| S3 / MinIO managed | Репликация в другой S3 | Другой S3 из пула |
| External S3 | Не наша ответственность | — |

Бэкап = snapshot приложения целиком (образ + данные + конфиг). БД и код не разделяются — гарантированная консистентность при восстановлении.

master.key хранится отдельно от архива бэкапа всегда.

### Расписание

Настраивается отдельно для CP и для каждого managed сервера.

### Retention

Шаблоны: 7д / 30д / 90д / кастом (количество копий или период).

### Валидация

CP автоматически проверяет каждый бэкап (контрольная сумма, возможность распаковки). Алерт если бэкап повреждён или отсутствует по расписанию.

### Восстановление

Через UI: выбрал дату, выбрал сервер, нажал «Восстановить». Восстановление только целиком (приложение + данные). Частичное восстановление отдельного модуля — не поддерживается.

## 10. Управление за ОС *(не MVP)*

Разделение типов серверов:
- `docker-only` — CP управляет только Docker-стеком
- `full` — CP следит за ОС: обновления пакетов, безопасность, мониторинг ресурсов

При добавлении `full`-сервера CP обновляет ОС, ставит нужные пакеты, настраивает окружение.

---

## 11. Установочный скрипт CP

`curl install.sh | bash` делает:
1. Проверяет ОС и архитектуру
2. Ставит Docker если не установлен
3. Обновляет ОС (опционально)
4. Скачивает `docker-compose.yml` CP
5. Генерирует `master.key`
6. Запускает CP
7. Открывает browser на wizard (или выводит URL)

## 13. MVP vs Не MVP

### В MVP

- Web UI + CLI
- Wizard первого запуска
- Server registry (docker-only + mock)
- Module registry + деплой + rollback
- Групповые операции
- Шаблоны инсталляций
- License manager (через Лицензикон API)
- Update watcher (ручное применение)
- Notification Hub (Telegram + Max + Email)
- S3 пул (managed MinIO + external) + файловый менеджер
- Backup (CP + managed серверы)
- Мониторинг + диагностика + панель
- Аудит лог
- SSL (Let's Encrypt HTTP-01 + custom + ГОСТ)
- Hardening чеклист при добавлении сервера
- Статус-страница + баннер в LMS
- Maintenance page при деплое
- Авторизация (local + OIDC + 2FA + IP whitelist)
- Telegram Bot + Mini App (мониторинг)
- Mobile-адаптивный UI
- Pre-flight checks (минимум + рекомендованные)
- Расписание операций
- История операций с логами
- Конфигурация модулей через форму + drift detection

### Не MVP (в планах)

- full-тип сервера (управление ОС)
- Webhooks из CP
- Квоты по организациям
- DR (автоподъём на резервном сервере)
- Disaster Recovery

## 16. Экосистемный контур: Ops-plane vs Product-plane

CP — инструмент для IT-персонала (SSH-ключи, секреты, деплой), может сидеть за VPN/IP-whitelist. Лаунчер для конечных пользователей (студенты, преподаватели) — отдельная сущность: смотрит в интернет, должен быть доступен всегда, для тысяч пользователей. Смешивать периметры нельзя: компрометация публичного лаунчера не должна открывать путь к SSH-ключам всех серверов, а простой CP ради своего релиза/бэкапа не должен ронять доступ студентов к занятиям.

```
Ops-plane (IT, может быть за VPN/IP-whitelist)
┌────────────────────────┐
│  CP                     │
│  SSH · секреты          │
│  лицензии · бэкапы      │
└───────────┬─────────────┘
            │ деплоит и конфигурирует как обычный модуль
            ▼
Product-plane (всегда доступно всем пользователям)
┌─────────────────────────────────────────┐
│  Лаунчер  ── OIDC вход                    │
│       ▼                                   │
│  Дидактикон · Кабиторий · Хроникон · ... │
│       ▼                                   │
│  Шлюз Универкона (per-instance sidecar)   │
│       ▼                                   │
│  Универкон Core (держатель истины)        │
└─────────────────────────────────────────┘
```

CP деплоит Лаунчер и Шлюз так же, как остальные модули — через `module.yaml` + docker compose. Физически они не CP.

**Лаунчер получает данные из двух источников разной свежести:**
- От CP при деплое — манифест «какие модули установлены + брендинг», обычный Class C push
- От Шлюза в реальном времени — что доступно конкретному пользователю (роль, права). Не зависит от того, работает ли CP прямо сейчас

### 16.1 Шлюз Универкона

**Единица развёртывания — один экземпляр Универкон Core, не организация.**
У организации может быть несколько инстансов Core одновременно (`environment: prod | staging | dev`, уже в схеме сервера, см. 5.1) — у каждого свои данные. Шлюз садится сайдкаром 1:1 к каждому запущенному Core: иначе staging-фронт через общий шлюз рискует достучаться до прод-данных.

**Не бандлится с `univerkon-core`.** Core работает самостоятельно и без единого фронта — шлюз ему для собственной работы не нужен. Отдельный опциональный модуль, деплоится когда есть хотя бы один тонкий фронт с браузерным логином.

**Роль — версия и маршрутизация, не трансляция.**
Шлюз не прячет рассинхрон версий трансляцией схем — это копит скрытый долг и не даёт видеть реальное состояние системы. Вместо этого: честная проверка версии контракта, явный отказ с понятной ошибкой если Core отстал от того, что требует фронт. Обязанность «догонять» лежит на релизном цикле Универкона, не на шлюзе.

Это даёт конкретный механизм для уже зафиксированной pre-flight-проверки «версии зависимостей совместимы» (см. 5.3) применительно к связке фронт↔шлюз↔Core.

```
Универкон Core                              — deps: []
Шлюз                                        — deps: [univerkon-core]
Фронт с браузер-логином (Дидактикон и др.)  — deps: [univerkon-core, шлюз]
Безфронтовый сервис (Тестикон)              — deps: [univerkon-core]   ← как в примере module.yaml, 5.2
```

Тестикон не имеет собственного браузерного UI (тонкий клиент — Дидактикон), поэтому обращается к Core напрямую сервис-к-сервису, минуя Шлюз.

Дополнительные обязанности Шлюза: OIDC-релей (единая точка входа с любым провайдером — Универкон или сторонний), Outbox/Inbox для Кабитория и Хроникона вместо повторной реализации в каждом модуле.

**Имя Шлюза и Лаунчера** — не приоритет сейчас. По конвенции `-кон`, вероятно, получат свои имена позже.

### 16.2 Мок vs Демо vs Dev-заглушка

Три разных механизма несмотря на похожие названия:

| Что | Когда используется | Реализация |
|---|---|---|
| Mock-сервер CP | Демо/обучение самого CP | Фейковый SSH-таргет, контейнеров нет вообще |
| Dev-заглушка | Универкон/фронт ещё не готовы, а модуль уже пишется | Фейковый API, отвечает как настоящий |
| Демо-режим | Реальный задеплоенный модуль без лицензии | Настоящий контейнер и БД, урезан по `module.yaml` (см. 5.4) |

Общее между ними — не механизм, а данные: один пакет фикстур (примеры курсов, тестов, студентов) наполняет и dev-заглушку, и демо-режим, и mock CP для презентаций. Выносится shared-пакетом (см. 16.3).

### 16.3 Общие пакеты

Клиентов уже несколько (Дидактикон, Кабиторий, Хроникон, Росткон) плюс Лаунчер — без общего фундамента разъедутся визуально и по поведению даже с одинаковыми токенами брендинга.

```
Репозиторий: univerkon-ui        (pnpm workspaces, не монолит всего проекта)

Пакеты:
  @univerkon/tokens        — design tokens (цвет/шрифт/отступы), база под brand.json от CP
  @univerkon/ui            — UI-kit с доступностью по умолчанию
  @univerkon/auth          — OIDC-вход, истечение сессии, переключатель роль↔контекст, единый logout
  @univerkon/api-client    — обёртка над JSON-RPC 2.0
  @univerkon/fixtures      — пакет сид-данных для mock / dev-заглушки / демо-режима
```

pnpm workspaces + Changesets: свой семвер и changelog у каждого пакета, один репозиторий — потому что на этом этапе пакеты меняются синхронно и часто. Changesets нативно работает с Conventional Commits (см. раздел о процессе разработки).

Каждый продуктовый модуль остаётся в своём репозитории и подключает нужные версии `@univerkon/*` как обычные npm-зависимости — тот же принцип, что уже используется для `deps` в `module.yaml`, только теперь и на фронтенде.

## 17. Процесс разработки

Всё ниже — не специфика CP, а единый стандарт для всей экосистемы: CP, каждый -кон, Шлюз, Лаунчер, `univerkon-ui`. Механика применения — 17.9.

### 17.1 Репозитории и Git

- Poly-repo: каждый продукт — свой репозиторий (CP, каждый -кон, Шлюз, Лаунчер). Общий фронтенд-код — `univerkon-ui` (см. 16.3)
- Backlog объединяется через GitHub Projects (org-level доска) поверх Issues из всех репозиториев
- Commit convention — Conventional Commits (`feat`/`fix`/`docs`/`BREAKING CHANGE`)
- Changelog и версия — автоматически из коммитов через release-please (или git-cliff), вручную не редактируется
- Ветвление — trunk-based: `main` + короткоживущие feature-ветки, без GitFlow
- PR: обязательный review, обязательные зелёные CI-checks, прямой push в `main` закрыт
- CODEOWNERS — на случай внешних контрибьюторов

### 17.2 Лицензирование

- **Код:** AGPL-3.0. Сетевое использование (SaaS-хостинг форка) обязывает открыть изменения — защищает именно от кейса «кто-то захостит наш CP как платный сервис и не поделится». Прецедент в собственном стеке — MinIO живёт на той же лицензии
- **Контент** (книги, edu-framework) — остаётся CC BY-SA, это отдельная лицензия от кода
- **Dual-license:** как правообладатель Александр сохраняет право продавать коммерческую лицензию без AGPL-обязательств отдельным клиентам. Для этого внешние контрибьюторы подписывают DCO (`git commit -s`) при каждом коммите — минимальный барьер, сохраняющий за проектом право на релицензирование. Полноценный CLA — только если понадобится более сильная защита, решается при первом внешнем PR
- Модули (-коны) — та же лицензия для консистентности; реальная защита монетизации у них не в лицензии кода, а в закрытом бэкенде выдачи лицензий (Лицензикон)

### 17.3 CI/CD

Минимум на компе разработчика — весь тяжёлый процесс в GitHub:

- **Codespaces / devcontainer.json** — окружение (Go, Node, линтеры) поднимается в браузере, локально ничего не ставится
- **На PR:** lint → test → build (без публикации образа)
- **На merge в main:** то же + публикация образа с тегом `edge`
- **На тег `vX.Y.Z`:** multi-arch build (amd64/arm64) → публикация образа с semver-тегом → changelog из Conventional Commits → GitHub Release
- Линтеры (`golangci-lint`, `eslint`) — gate в CI, не pre-commit hook

### 17.4 Тестирование и покрытие

- Интерфейс покрывается тестами наравне с бэкендом, включая доступность (a11y)
- Mock-сервер (уже часть продукта) — готовый e2e-fixture: Playwright гоняет полный сценарий («добавил сервер → задеплоил модуль → проверил статус») без реального SSH и Docker — дёшево и стабильно в CI
- **Coverage gate — patch/diff, не глобальный** (через Codecov/Coveralls, бесплатны для OSS):
  - 70% нового кода в PR — по умолчанию
  - 90% — для критичных пакетов: `internal/secrets`, `internal/auth`, `internal/license`
  - без строгого числа — TUI-рендер и визуальный React; полагаемся на e2e через mock-сервер
- Общий процент по репозиторию — виден бейджем в README, но сам по себе не блокирует мердж

### 17.5 Документация

- Весь код и проект покрыты документацией
- Сайт — GitHub Pages, собирается из статики (MkDocs Material)
- При poly-repo — агрегатор: отдельный репозиторий `univerkon-docs`, на CI подтягивает `docs/` из каждого продуктового репозитория и собирает единый сайт. Рабочий дефолт, меняем если неудобно на практике
- **ADR** (Architecture Decision Records) — тот же принцип что уже работает в инвариантах Тестикона и Дидактикона, только для технических решений CP: `docs/adr/0001-secrets-encryption.md`
- **OpenAPI-спека** — автогенерируется из кода, рендерится на том же сайте
- **Доступность (a11y)** — войдёт в Definition of Done для UI-задач по получении документа

### 17.6 Безопасность процесса

- `SECURITY.md` — куда и как сообщать об уязвимостях, отдельно от публичных Issues. Периметр чувствительный: SSH-ключи, секреты, лицензии медицинских учреждений
- Dependabot/Renovate — автообновление зависимостей с security-патчами, особенно для Go-модулей криптографии (SSH, AES-GCM)

### 17.7 Backlog-дисциплина

- Любая правка — только через Issue, без исключений
- Issue templates: bug / feature / question
- PR template с обязательной ссылкой на Issue — без неё PR не мержится
- **Definition of Done:** тесты + документация + changelog entry + a11y-чек для UI-задач + пройденный review

### 17.8 Прочее

- **Deprecation policy** для `module.yaml` / API Шлюза: минимум один minor-релиз в статусе deprecated (предупреждение в логах CP и в доках) перед удалением в следующем major
- **Observability CP** — структурированные JSON-логи для отладки CP на стороне клиента, отдельно от продуктового мониторинга managed-серверов (см. 5.8)
- **Оформление репозитория:** бейджи в README (build status, license, coverage), корневой README минимальный, дальше всё по ссылкам

### 17.9 Механика применения ко всей экосистеме

Стандарт документируется один раз — применяется механически, не копипастой в каждый репозиторий (копия неизбежно расходится через полгода):

- **`.github`-репозиторий на уровне организации.** GitHub подхватывает `SECURITY.md`, `ISSUE_TEMPLATE/`, `PULL_REQUEST_TEMPLATE.md`, `CODE_OF_CONDUCT.md` из репозитория с именем ровно `.github` как дефолт для любого репо организации, у которого нет своей копии
- **Reusable workflows** (`workflow_call`) — пайплайн lint→test→build→release написан один раз в центральном месте, каждый продуктовый репозиторий подключает его тонким файлом в 5 строк. Меняешь пайплайн — меняется сразу везде, без ручной синхронизации по репозиториям
- **Template repository** — новый -кон стартует не с нуля, а «Create from template»: devcontainer, CI, линтеры, issue-темплейты уже на месте. Тот же принцип, что уже работает у `imiron-static-site-template`, просто применённый к экосистеме Универкон
- **Расшаренные конфиги линтеров** — `@univerkon/eslint-config` пакетом (рядом с `@univerkon/ui` и остальными в 16.3), общий `.golangci.yml` через template repository

**Что не переносится один в один, а передумывается для каждого репозитория:** список «критичных пакетов» под 90%-coverage (`internal/secrets` осмысленно для CP, но не для Хроникона — там свой список), и конкретные шаги CI под язык/тип артефакта (Go-бинарник vs npm-пакет vs Docker-образ).

Поскольку `control` (репозиторий CP, org `aplicon-ru`) — первый реальный репозиторий экосистемы, `.github`-организацию и reusable workflows выносить пока не во что: они пишутся прямо внутри `control` за этот заход, но так, чтобы вынести в отдельные репозитории было чисто, когда появится второй продукт.
